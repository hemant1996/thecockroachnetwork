# v0.2 design — WebRTC peer relays in the browser

> **Status:** design sketch, not yet implemented. Targeted for v0.2.

## The gap this fixes

Today, running a relay requires hosted infrastructure — a VPS, a Render account, a Replit workspace, or Termux on an Android device. Even the easiest of these (Render, Replit) requires creating an account on a third-party platform. For "0 control from day 1" to be true *in practice* and not just *in theory*, the lowest possible bar to relaying must be: **open the client in a browser, and your device is part of the network.**

A browser tab cannot accept inbound WebSocket connections — that is an OS- and browser-level security constraint that no amount of cleverness defeats. The correct architecture for in-browser relays is **WebRTC peer-to-peer**: every PWA install becomes a peer in a mesh, no inbound port required, no hosted server, no DNS.

## Architecture

Three layers, of which only the WebRTC mesh is new:

```
┌─────────────────────────────────────────────────────────┐
│ traditional relays  (WebSocket, fixed URL, persistent)  │  ← already in v0.1
│   wss://relay.example.org                               │
└──────────────────┬──────────────────────────────────────┘
                   │  bridges
┌──────────────────▼──────────────────────────────────────┐
│ WebRTC peer mesh  (browser ↔ browser, ephemeral)        │  ← v0.2
│   each PWA install = one peer                           │
└─────────────────────────────────────────────────────────┘
```

Traditional relays remain the durable backbone — they hold the 90-day retention store, serve OG-tagged permalink pages, and act as bootstrap nodes. The WebRTC mesh is a complement: it absorbs traffic spikes, survives the loss of any individual relay, and lets devices in a region (a protest, a neighborhood) gossip directly even when their upstream connection is degraded.

## Wire protocol additions

Three new ephemeral event kinds in the `10000+` range (per [SPEC §4.3](../SPEC.md#43-reserved-kind-ranges)). Ephemeral means: relays MAY forward them but SHOULD NOT store them long-term.

### `kind: 10001` — peer offer

A peer's WebRTC connection offer, broadcast via traditional relays so other peers can find it.

```json
{
  "kind": 10001,
  "tags": [
    ["sdp", "<base64 SDP offer>"],
    ["g",   "<geohash>"],        // optional, for locality-based peer selection
    ["expires", "<unix timestamp>"]
  ],
  "content": ""
}
```

### `kind: 10002` — peer answer

A peer's WebRTC answer, addressed to a specific offerer.

```json
{
  "kind": 10002,
  "tags": [
    ["p",   "<offerer-pubkey>"],
    ["e",   "<offer-event-id>"],
    ["sdp", "<base64 SDP answer>"]
  ],
  "content": ""
}
```

### `kind: 10003` — ICE candidate

Trickle-ICE candidates for an in-progress connection.

```json
{
  "kind": 10003,
  "tags": [
    ["p",   "<peer-pubkey>"],
    ["e",   "<offer-event-id>"],
    ["ice", "<base64 ICE candidate>"]
  ],
  "content": ""
}
```

## Peer behavior

A peer-enabled client behaves as follows on boot:

1. Connect to traditional relays as today.
2. Publish a `kind:10001` offer with a 1-hour expiry, tagged with the user's geohash (precision 5 by default — coarse, for privacy).
3. Subscribe to `kind:10001` events filtered by `#g` near the user's geohash, plus `kind:10002` and `kind:10003` tagged to the user's pubkey.
4. For each matching offer, generate a WebRTC answer and publish `kind:10002`. Open the peer connection.
5. Exchange ICE candidates via `kind:10003`.
6. Once the peer connection is up, gossip events bidirectionally over the WebRTC data channel using the same canonical event format. Use a Bloom filter or recent-id set to avoid re-broadcasting.

The browser-peer is **not** a permanent relay — when the tab closes, the peer goes away. Other peers detect the dropped data channel and prune their peer table.

## Bootstrap and discovery

Discovery uses the traditional relay layer, not a separate DHT. This keeps the design simple and means the WebRTC mesh inherits the censorship-resistance properties of the relays it bridges.

If all traditional relays a client knows about go offline simultaneously, **a peer-enabled client that already has at least one live WebRTC peer can stay on the network** — that peer can re-broadcast any future events. New peers cannot join the mesh until at least one traditional relay comes back up to carry the signaling events. This is an acceptable degradation: the system survives short outages naturally and recovers as soon as a single relay reappears.

## NAT traversal

Most browsers are behind NAT. WebRTC handles this via STUN (always-works for symmetric NAT detection and address discovery) and TURN (relay fallback for hostile NATs). For v0.2:

- **STUN:** use Google's public STUN server (`stun:stun.l.google.com:19302`) as the default. Trivially replaceable in client config; operators of mirrors can ship their own STUN list.
- **TURN:** NOT shipped by default. TURN servers cost bandwidth and represent a centralization risk. Users who cannot establish direct WebRTC due to symmetric NATs simply do not join the mesh; they continue to use traditional relays. This is a fully acceptable degradation.

A community of operators may run open TURN servers for users in hostile NAT environments; the client lists them as a configurable list, identical to the relay seed-list pattern.

## Privacy considerations

WebRTC reveals more than WebSocket-to-relay does:

- **Direct peer-to-peer connections expose IP addresses to other peers**, not just to the relay you publish through.
- Mitigation: peer-enabled mode is **off by default** in v0.2. Users opt in via a clear toggle in the Identity tab with copy explaining the IP exposure.
- Mitigation: a "Tor-only" mode disables WebRTC entirely (Tor and WebRTC do not mix well; users routing over Tor should not be peer-relaying).
- The geohash tag on `kind:10001` defaults to precision 5 (~5 km), not the precision 7 used for civic reports. Coarsening location for peer discovery prevents WebRTC from being a more precise location-leak channel than the report itself.

## Storage

Each peer stores recently-seen events in IndexedDB (default cap 50 MB, ~100,000 events at 500 bytes each). When the cap is hit, the oldest events are evicted. Storage is per-pubkey-namespace, not global — different identities share the database via a key prefix.

A peer serves stored events to newly-connected peers asking via the WebRTC channel with a filter (same wire syntax as `REQ`). This lets a fresh peer warm-start without needing a traditional relay, *if* it can reach an existing peer.

## What this does NOT promise

- **Replacement of traditional relays.** A pure-WebRTC network has no permanent storage, no permalinks with OG tags, no operator-policy enforcement, and no easy bootstrap. v0.2's mesh complements the relay layer; it does not replace it.
- **Mobile background operation.** A backgrounded iOS Safari or Chrome tab is throttled aggressively; peer connections drop. The mesh is most useful while the app is in the foreground or while the PWA is open on a screen-on phone or a laptop.
- **Defense against well-resourced adversaries.** A coordinated state-level actor can identify peers (IPs exposed), correlate them, and stand up sybil peers in the mesh. The locality + reputation defenses from v0.1 (SPEC §8) still apply at the event-evaluation layer, not at the peer-membership layer.

## Implementation plan

When v0.2 work begins, the order is:

1. Library selection: evaluate `simple-peer` vs. `libp2p-js`. Lean toward `simple-peer` for size (~30 KB minified) and minimal API surface.
2. Add WebRTC peer manager to `client/app.js` behind a feature flag.
3. Extend `RelayPool` to broadcast outbound events to peer connections as well as relays.
4. Implement signaling event handlers (`kind:10001/10002/10003`) in the relay reference implementation — at the protocol level these are just ephemeral events with `kind >= 10000`, no code change should be required for compliant relays.
5. Build the Identity-tab toggle, IP-exposure consent flow, and STUN/TURN configuration UI.
6. Integration test with three browsers on three NATs.

Estimated effort: 1–2 weeks for a working PoC, 4–6 weeks for production-ready release.
